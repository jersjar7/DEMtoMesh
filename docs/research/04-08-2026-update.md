### Stage 5b: Qwen2.5-VL Grounding Test (Failed — Useless Bounding Boxes)
**Notebook:** `03_qwen25vl_grounding_test.ipynb`
**Model:** `qwen/qwen2.5-vl-72b-instruct` via OpenRouter

Qwen2.5-VL was asked to output JSON bounding boxes for terrain features. Results:

| Test | Bbox Returned | Actual Meaning |
|------|--------------|----------------|
| hillshade_channel | `[2, 0, 1036, 588]` | Entire image |
| hillshade_ridges | `[0, 0, 1036, 588]` | Entire image |
| hillshade_floodplain | `[0, 0, 1036, 588]` | Entire image |
| hillshade_manmade | `[1, 0, 1035, 588]` | Entire image |
| slope_channel | `[378, 298, 676, 500]` | Plausible but imprecise |
| curvature_channel | `[360, 297, 654, 480]` | Plausible but imprecise |
| elevation_channel | `[0, 0, 1036, 588]` | Entire image |

**Root cause:** Qwen2.5-VL was trained for object detection on natural photographs (people, cars, animals). When asked to localize abstract terrain features in a grayscale hillshade, it defaults to boxing the entire image. The model *understands* the features (its qualitative survey was decent) but cannot *localize* them spatially.

**Mitigations attempted:**
- Aligned image resize to multiples of 28 (Qwen's internal grid) — did not help
- Used concrete visual descriptions instead of abstract geomorphic terms — marginal improvement on slope/curvature only
- Added whole-image bbox filter (>90% area) — correctly filters junk but doesn't produce better results

**Verdict:** API-based VLM grounding is not precise enough for terrain features. These models need discrete, high-contrast objects to localize, not continuous landscape features.

### Stage 5c: Florence-2 Local Detection (Failed — Transformers Compatibility)
**Notebook:** `04_florence2_local_test.ipynb`
**Model:** `microsoft/Florence-2-large` (local, MPS)

**First attempt failed** with:
```
'Florence2LanguageConfig' object has no attribute 'forced_bos_token_id'
```

**Root cause:** `transformers==5.5.0` removed generation parameters from `PretrainedConfig`. The `microsoft/Florence-2-large` checkpoint uses custom remote code that references `self.forced_bos_token_id`, which no longer exists on the config object.

**Fix applied:** Switched to `florence-community/Florence-2-large` with the native `Florence2ForConditionalGeneration` class (no `trust_remote_code` needed). This is the officially recommended path for transformers >= 5.0.

**Second run results:** Florence-2 loaded successfully but returned **entire-domain bounding boxes** for all tasks — same fundamental problem as Qwen. Object detection (`<OD>`), phrase grounding (`<CAPTION_TO_PHRASE_GROUNDING>`), and dense region captioning (`<DENSE_REGION_CAPTION>`) all produced boxes covering most or all of the image.

**Verdict:** Florence-2 was trained on natural imagery. DEM renderings are too far outside its training distribution for meaningful spatial localization.

## 2. Key Findings

### What Works
1. **VLMs can identify terrain features qualitatively.** Gemma 4 (and Qwen's qualitative mode) correctly describe channels, floodplains, ridges, embankments, etc. from DEM renderings. The visual intelligence is there.
2. **Multiple DEM renderings provide complementary information.** Curvature was best for subtle features (field boundaries, ditches). Slope was best for channel banks. Hillshade gave the richest overall descriptions.
3. **The pipeline concept is sound.** The issue isn't the architecture — it's the input data. We're asking models to localize objects in images that look nothing like their training data.

### What Doesn't Work
1. **API-based VLM grounding on DEM imagery.** No model tested (Qwen2.5-VL, Florence-2) can produce useful bounding boxes for terrain features in hillshade/slope/curvature images.
2. **Abstract geomorphic terms as detection targets.** Asking a model to "detect the floodplain" doesn't work — floodplains aren't discrete objects. They're continuous, diffuse landscape regions.
3. **DEM-only input for spatial localization.** The models were never trained on terrain derivatives. A hillshade looks like a grayscale photograph of nothing recognizable to them.

### The Fundamental Problem
VLMs are trained on **natural photographs**. They know what rivers look like in satellite imagery — water, vegetation boundaries, sediment patterns. They do NOT know what rivers look like in a hillshade — a gray-on-gray sinuous depression with subtle shading.

We were asking the models to recognize features in an alien visual domain.

## 3. The Pivot: Dual-Input Pipeline (Satellite + DEM)

### The Insight
Instead of forcing VLMs to interpret terrain derivatives, give them what they understand: **satellite imagery**. Then use the DEM for what it's actually good at: **precision**.

### Revised Pipeline

```
OLD PIPELINE (failed):
  DEM → Hillshade → VLM (identify + localize) → SAM → Mesh

NEW PIPELINE (proposed):
  Satellite Imagery → VLM (identify + localize features)
       ↓                    ↓
  Points/BBoxes         Text descriptions
       ↓                    ↓
  SAM on Satellite     LLM reasoning
  (rough masks)        (mesh specs)
       ↓
  DEM Derivatives → Refine masks to precise breaklines
  (slope, curvature)   (snap to elevation discontinuities)
       ↓
  Georeferenced feature map
       ↓
  Mesh generation (Gmsh)
```

### Why This Should Work
1. **Satellite imagery is in-distribution.** VLMs have seen millions of aerial/satellite photos during training. Rivers look like rivers. Roads look like roads. Vegetation boundaries are clear.
2. **DEM provides sub-meter precision.** Satellite imagery tells you WHERE the channel is; the DEM curvature map tells you exactly WHERE the bank edges are to the nearest pixel.
3. **Complementary strengths:**
   - Satellite: feature recognition, classification, rough localization
   - DEM hillshade: fine geometry, breakline tracing, bank delineation
   - DEM slope: channel bank edges (high slope), floodplain boundaries
   - DEM curvature: breaklines (high curvature = sharp elevation change)
4. **Multi-image prompts.** Modern VLMs (Qwen2.5-VL, Gemini, Claude) support multiple images per query. We can send satellite + hillshade together: "the dark sinuous feature in image 1 (hillshade) is the river visible in image 2 (satellite) — trace its banks precisely."

### Satellite Imagery Source: NAIP via Microsoft Planetary Computer

**NAIP (National Agriculture Imagery Program):**
- **Resolution:** 0.6m per pixel — actually HIGHER than our 1m DEM
- **Bands:** Red, Green, Blue, Near-Infrared (4-band)
- **Coverage:** Entire continental US, updated every 2-3 years
- **Cost:** Free, no API key required
- **Access:** Microsoft Planetary Computer STAC API with anonymous tokens
- **Oklahoma coverage:** Confirmed available (2021 and likely 2023)

**Python packages needed:**
```
pip install pystac-client planetary-computer rioxarray
```

**Why NAIP over alternatives:**

| Source | Resolution | Why Not |
|--------|-----------|---------|
| **NAIP** | **0.6m** | **Winner — free, high-res, easy API** |
| Sentinel-2 | 10m | 16x coarser than NAIP — too low-res for feature-level detection |
| USGS HRO | 0.15-0.3m | Coverage is spotty, requires M2M approval, not guaranteed for this site |
| Mapbox | ~0.5m | TOS prohibits bulk download for offline analysis |

**Download approach:** Query the Planetary Computer STAC catalog for NAIP tiles intersecting our DEM bounding box (converted to WGS84: `(-97.39, 35.98, -97.33, 36.00)`), download as Cloud-Optimized GeoTIFF, clip to DEM extent + buffer.

## 4. Molmo 2 — Still the Best Candidate for Coordinate Grounding

Molmo 2 remains the ideal model for this pipeline. Here's why:

| Capability | Molmo 2 | Qwen2.5-VL | Florence-2 |
|------------|---------|-------------|------------|
| Point coordinates | Native (`<point x y>`) | Not native (JSON hack) | Not supported |
| SAM prompt type | Direct point prompt | Bbox (less precise) | Bbox (less precise) |
| Training data | Natural images + pointing annotations | Natural images | Natural images |
| Runs locally | Yes (7B, fits on M1 Pro 32GB) | No (72B, API only) | Yes (0.7B) |
| API available | Not yet on OpenRouter | Yes | N/A (local only) |

**Paths to get Molmo 2 running:**
1. **Ollama** — `ollama pull molmo2` (if available)
2. **HuggingFace transformers** — `allenai/Molmo-7B-D-0924` or newer Molmo 2 checkpoint
3. **vLLM** — local API server mimicking OpenRouter interface

With satellite imagery as input, Molmo 2's pointing capability becomes extremely powerful: "Point to where the river channel bends" → exact pixel coordinates → SAM point prompt → precise segmentation mask.


### Qwen2.5-VL Coordinate System
- Coordinates are **absolute pixels** relative to the model's internal resize (multiples of 28)
- NOT the 0-1000 normalized scale used by Qwen2-VL (the previous generation)
- Pre-aligning images to 28px multiples helps coordinate accuracy
- Through OpenRouter, you cannot control the model's internal resize or pass `resized_width`/`resized_height` metadata

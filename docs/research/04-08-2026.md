## 1. The Problem

Currently, hydraulic software programs (e.g., SMS, HEC-RAS 2D) require engineers to spend most of their time building the best mesh to represent the terrain for the model to run. A skilled modeler might spend 60–80% of their project time on mesh construction alone. The workflow is painfully manual:

1. Import a DEM
2. Visually identify features (channels, floodplains, ridges, levees, road embankments)
3. Draw arcs and polygons to define mesh regions
4. Assign element sizes
5. Set breaklines
6. Choose quads vs. triangles
7. Refine, run, check results, refine again

## 2. The Core Idea

Since AI is great at recognizing features from an image (i.e., a DEM rendered as a hillshade), can we use it to generate the size of the mesh elements depending on the features (i.e., quads for a channel, breaklines on ridges, etc.) and therefore the mesh for the entire domain of the study?

**Key insight from Jerson:** What if we don't need to train any model at all? We can use the computer vision capabilities that LLMs currently excel at, match those features with DEM values per cell, and then use LLM reasoning to decide what type of mesh feature to place at every location in the domain.

This approach skips the entire training data problem. No need for thousands of labeled DEM-mesh pairs. It leverages the fact that LLMs already understand what channels, ridges, and floodplains look like from pretraining, and already have hydraulic engineering domain knowledge.

**Dr. Zundel (Jerson's class instructor), the creator of SMS, knows this has not been done yet.** This would be breakthrough technology in the hydraulic modeling world.

## 3. Why This Is Novel

- People have used traditional computer vision (edge detection, feature extraction algorithms) for mesh generation assistance
- There's academic work on ML-based mesh refinement
- But using a foundation model's *general visual reasoning and domain knowledge* to drive the full mesh specification — essentially replacing the engineer's visual assessment and judgment calls — has not been published or productized
- This approach is enabled by capabilities that have only existed for about two years

## 4. The Proposed Pipeline

```
Step 1: Load DEM (GeoTIFF)
    ↓
Step 2: Render as hillshade / slope map / curvature map
    ↓
Step 3: Feed image to vision models (Molmo, Florence-2)
        → Models identify hydraulic features and return coordinates
    ↓
Step 4: Feed coordinates to SAM/SamGeo
        → Pixel-level segmentation masks for each feature
    ↓
Step 5: Map segmented regions back to DEM cells
        → Cross-reference with actual elevation values
    ↓
Step 6: LLM reasoning (Claude/GPT-4o) decides mesh specs
        → Element type, size, alignment per region
    ↓
Step 7: Feed specification to meshing engine (Gmsh)
        → Generate actual computational mesh
    ↓
Step 8: Export to SMS (.2dm) or HEC-RAS format
```

**Design philosophy:** "The AI does 80% of the work in minutes instead of hours; the engineer reviews, adjusts, and approves." We turn stochasticity from a bug into a feature.

## 5. Research Findings — Models & Tools

### 5.1 Molmo / Molmo 2 (Allen Institute for AI)

- **What it does:** Returns precise (x,y) pixel coordinates from natural language prompts ("point to the channel")
- **Key detail:** Outputs point coordinates as normalized values between 0 and 100, which rescale to original image dimensions
- **Why it matters for us:** Designed specifically for "pointing" — ask it to identify a feature and it gives you coordinates, not just a description
- **Molmo 2 (Dec 2025):** 8B model outperforms original 72B on pointing/grounding benchmarks
- **Already used for:** 3D keypoint detection — researchers used Molmo to localize 2D keypoints, then back-projected into 3D using depth data (conceptually identical to our DEM use case)
- **Open source:** Full weights, datasets, and code available
- **Links:**
  - https://allenai.org/blog/molmo2
  - https://arxiv.org/html/2409.17146v2
  - https://learnopencv.com/molmo-vlm/

### 5.2 Florence-2 (Microsoft)

- **What it does:** Lightweight vision-language model for object detection, grounding, and segmentation
- **Key detail:** Outputs bounding boxes (x1,y1,x2,y2), quad boxes (4 corners), and polygon vertices for segmentation
- **Why it matters for us:** Can identify features by name ("channel", "ridge") and return polygon coordinates tracing their boundaries
- **Trained on:** FLD-5B dataset — 126 million images, 5.4 billion annotations
- **Zero-shot capable:** Works without fine-tuning
- **Open source:** MIT license, available on Hugging Face
- **Links:**
  - https://huggingface.co/docs/transformers/en/model_doc/florence2
  - https://roboflow.com/model/florence-2

### 5.3 SAM / SamGeo (Meta + Geospatial Community)

- **What it does:** Pixel-level segmentation from minimal prompts (point, bounding box, or text)
- **SamGeo specifically:** Python package that applies SAM to georeferenced GeoTIFF files, handles tiling of large rasters
- **Key detail:** Already integrated with ArcGIS and QGIS
- **Why it matters for us:** Takes the approximate locations from Molmo/Florence-2 and produces precise, pixel-level masks
- **Performance:** Point-based prompts outperformed other approaches for distinct objects
- **Links:**
  - https://samgeo.gishub.org/
  - https://docs.ultralytics.com/models/sam/

### 5.4 TASAM — Terrain-Aware SAM

- **What it does:** SAM variant enhanced with DEM/terrain priors
- **Key detail:** Includes a "Terrain-Aware Adapter" that injects topographic priors from DEMs into the SAM image encoder
- **Why it matters for us:** Someone has already built a SAM that understands terrain! Shows +6.5 mIoU improvement on change-sensitive segmentation, and 10+ mIoU gains on small-object clusters
- **Link:** https://arxiv.org/html/2509.15795

### 5.5 Florence-2 + SAM Pipeline (Already Exists)

- **Key finding:** People are already chaining Florence-2 → SAM in production
- **Florence2toCoordinates** node converts Florence-2 bounding boxes into coordinate formats SAM can consume
- **Implication:** The "identify with language, segment with precision" pipeline is proven — we just need to apply it to terrain

### 5.6 LLM Reasoning Layer (Claude / GPT-4o)

- **Role:** Takes feature classifications + DEM statistics and reasons about mesh parameters
- **Not just "this is a channel" but:** "This channel is 15m wide, the floodplain is 200m wide, so use 3m quad elements aligned with flow direction, transitioning to 15m triangles on the floodplain, with 2:1 maximum transition ratio"
- **Key advantage:** These models have absorbed hydraulic engineering literature and meshing best practices during pretraining
- **Enhancement:** Provide meshing rules from Dr. Zundel as prompt context

### 5.7 Additional Relevant Research

- **PixelLM:** Pixel-level reasoning with large multimodal models — produces segmentation masks directly from LLM reasoning
- **PlotExtract:** Demonstrated that vision LLMs can extract precise spatial data from images with ~5% positional error, zero-shot
- **Foundation Models for Remote Sensing:** Benchmarked Molmo on Earth Observation tasks — effective when objects are sufficiently large and distinct

## 6. Four Key Challenges & Status

### Challenge 1: Spatial Precision
- **Status: Largely solved**
- Molmo → normalized coordinates → pixel positions
- Florence-2 → bounding boxes and polygon vertices
- SAM/SamGeo → pixel-level masks on georeferenced rasters
- **Remaining risk:** Do these models recognize terrain features in hillshade images? (Trained on natural images, not DEMs)
- **Mitigation:** Try multiple renderings (hillshade, slope, curvature, multi-band composites, vertical exaggeration)

### Challenge 2: Scale and Resolution
- **Status: Partially solved**
- Typical study domain: 10km × 10km at 1m resolution = 10,000 × 10,000 pixel raster
- Vision models work with ~1024×1024 images → need tiling
- SamGeo already handles tiling of large geospatial rasters
- **Remaining risk:** Edge consistency between tiles when features cross boundaries
- **Edge case:** Long meandering channel crossing multiple tiles at oblique angles

### Challenge 3: Consistency and Repeatability
- **Status: Solved by design**
- LLMs are stochastic — two runs may give different results
- **Solution:** Semi-automatic pipeline with human review
- AI generates a reviewable feature map → engineer approves → mesh is generated
- Low temperature settings reduce variance
- SAM is deterministic given the same prompts
- **Bonus:** Human-in-the-loop solves regulatory acceptance too (FEMA, USACE)

### Challenge 4: Domain-Specific Reasoning Quality
- **Status: Most uncertain — needs testing**
- Can an LLM make good mesh specification decisions with the right context?
- **Approach:** Provide DEM statistics per region + meshing rules from Dr. Zundel as prompt context
- LLM applies rules rather than guessing
- **Edge case:** Hydraulic structures (bridges, culverts, weirs) — poorly represented in DEMs, likely need manual handling
- **Edge case:** Subtle features (shallow swales, low levees) that may not be visually obvious in hillshade

## 7. Hydraulic Features to Detect

| Feature | Mesh Treatment | Visual Signature in DEM |
|---------|---------------|------------------------|
| Channel | Quad elements aligned with flow direction, fine resolution (2–5m) | Dark sinuous line in hillshade, high flow accumulation |
| Floodplain | Coarser triangles (15–30m), size based on terrain variability | Flat areas adjacent to channel |
| Ridge / Levee | Breakline to preserve sharp elevation change | Bright line in hillshade, high curvature |
| Road embankment | Breakline + local refinement | Linear feature, often perpendicular to channel |
| Bridge / culvert | Special treatment — likely manual | Gap in road embankment over channel |
| Flat storage area | Coarse elements (30m+) | Very flat, low slope |
| Steep hillslope | Medium elements, may exclude from active mesh | High slope values |

## 8. Tech Stack Decision

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Language | **Python** | Every tool in the pipeline is Python-native |
| IDE | **VS Code + Jupyter extension** | Interactive exploration + real project structure |
| Architecture | **Jupyter notebooks → modular package** | Prove concept fast, organize later |
| Vision models | **Local inference (GPU) or HF Inference API** | Control and speed for iteration |
| LLM reasoning | **Claude/GPT-4o API** | Text reasoning doesn't need local GPU |
| Mesh generation | **Gmsh Python API** | Open source, well-documented, flexible |
| Mesh I/O | **meshio** | Handles multiple mesh formats |

## 9. Development Roadmap

### Phase 1: Proof of Concept ← START HERE
> Can vision models see hydraulic features in a DEM?

- Load a small, well-understood DEM
- Render as hillshade, slope, curvature
- Feed to Molmo and/or Florence-2
- Ask to identify: channel, floodplain, ridges, roads
- Evaluate: do locations match reality?
- Feed identified points to SAM for precise segmentation
- Visualize everything

**This is the single highest-risk question.** If vision models can't distinguish a channel from a ridge in a hillshade, the approach needs rethinking. If they can, even roughly, we have a viable path.

### Phase 2: Mesh Specification
- Extract DEM statistics per segmented region
- Prompt LLM with feature data + meshing rules
- Get back element type, size, alignment recommendations
- Compare to expert-built mesh

### Phase 3: Mesh Generation
- Convert specs to Gmsh input
- Generate mesh, export to SMS/HEC-RAS format
- Run simple hydraulic simulation to validate

### Phase 4: Engineer Review Interface
- Streamlit or Panel GUI
- AI feature map overlaid on DEM
- Engineer adjusts and approves
- Re-generate mesh with adjustments
